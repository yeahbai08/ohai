
# 2. 完备性论证

## 2.1 方法论

要证明一组错误码能覆盖所有可能的失败场景，不能用归纳法（"参考别的协议看有什么码"），因为归纳永远无法证明穷尽。OHAI 采用 **演绎法**：将命令执行建模为一条有序管线，管线的每个阶段是命令成功的一个 **必要条件**。

**关键性质**：命令必须依次通过管线的所有阶段才能成功；命令在第一个失败的阶段停止并返回该阶段的错误码。因此，**任何失败必定发生在某个管线阶段**——只要每个阶段的失败模式被穷举，整体就是完备的。

## 2.2 管线阶段定义

```
命令到达设备
  │
  ▼
┌─────────────┐
│ ① 消息解码   │  消息能被解析为合法的 JSON-RPC 结构吗？
└──────┬──────┘
       ▼
┌─────────────┐
│ ② 命令路由   │  目标能力和命令在设备上存在吗？
└──────┬──────┘
       ▼
┌─────────────┐
│ ③ 参数校验   │  参数值对当前设备合法吗？
└──────┬──────┘
       ▼
┌─────────────┐
│ ④ 能力检查   │  设备支持此操作/模式吗？
└──────┬──────┘
       ▼
┌─────────────┐
│ ⑤ 前置条件   │  设备当前状态允许执行吗？
└──────┬──────┘
       ▼
┌─────────────┐
│ ⑥ 资源获取   │  执行所需的资源（计算资源 + 物理资源）都就绪吗？
└──────┬──────┘
       ▼
┌─────────────┐
│ ⑦ 物理执行   │  执行动作本身成功完成了吗？
└──────┬──────┘
       ▼
    命令成功
```

## 2.3 每个阶段的完备性分析

对每个阶段，我们枚举"该阶段可能失败的所有原因"，并为每个原因分配错误码。

## ① 消息解码

此阶段由 Server 在转发前完成，设备不涉及。失败原因恰好两种：消息不是合法的编码格式，或消息结构不符合 JSON-RPC 2.0 规范。

| 失败原因 | 错误码 |
|---|---|
| 消息不是合法的 CBOR/JSON | `PARSE_ERROR` |
| 消息结构不符合 JSON-RPC 2.0 | `INVALID_REQUEST` |

## ② 命令路由

Server 根据 `method` 字段定位目标能力和命令。失败原因恰好两种：目标不存在，或参数不符合 Schema。

| 失败原因 | 错误码 |
|---|---|
| 能力或命令名不存在 | `METHOD_NOT_FOUND` |
| 参数不符合命令的 JSON Schema | `INVALID_PARAMS` |

::: tip 阶段①②的说明
阶段①②的错误由 **Server 检测并生成**，不会到达设备。它们属于 JSON-RPC 2.0 标准保留错误码（`-32700` ~ `-32600`），此处列出是为了管线的完整性。Server 作为命令下发的中间层，在转发前已完成消息解码和路由校验。
:::

## ③ 参数校验（设备运行时）

参数通过了 Server 端的 Schema 校验，但设备在运行时可能因实际硬件约束拒绝。一个参数值被拒绝只有三种可能：**单值越界**（超出设备实际范围）、**单值精度不匹配**（范围内但步进/精度不被接受）、**多值组合矛盾**（各自合法但组合不兼容）。

| 失败原因 | 错误码 | 穷尽性说明 |
|---|---|---|
| 单值超出设备实际范围 | `VALUE_OUT_OF_RANGE` | 值与范围的关系：在范围外 |
| 单值精度/步进不被接受 | `VALUE_NOT_ACCEPTED` | 值与范围的关系：在范围内但不在离散可接受集合中 |
| 多值组合矛盾 | `INCOMPATIBLE_PARAMS` | 单值合法但组合不合法的唯一情况 |

## ④ 能力检查

参数合法，但设备本身是否支持这个操作。"不支持"只有两种粒度：**整个命令不支持**，或命令支持但**请求的特定模式/选项不支持**。此外还有一个特殊情况：命令被理解且支持，但**无需动作**（已在目标状态）。

| 失败原因 | 错误码 | 穷尽性说明 |
|---|---|---|
| 命令未实现 | `COMMAND_NOT_SUPPORTED` | 粗粒度：整个命令不支持 |
| 特定模式/选项不支持 | `MODE_NOT_SUPPORTED` | 细粒度：命令支持但某个模式值不支持 |
| 已在目标状态，无需动作 | `COMMAND_IGNORED` | 命令支持、参数合法，但执行等同于空操作 |

## ⑤ 前置条件

设备支持该命令，但**当前状态**不允许执行。"状态不允许"可以按阻碍的来源穷举——来自**设备自身状态**、来自**外部环境/物理世界**、或来自**其他并发操作**：

| 失败原因 | 错误码 | 穷尽性说明 |
|---|---|---|
| 设备处于冲突的工作模式 | `INVALID_STATE` | 设备自身状态：已就绪但模式冲突 |
| 设备尚未完成启动/初始化 | `NOT_READY` | 设备自身状态：未进入可工作状态（临时） |
| 设备未完成初始配置 | `NOT_CONFIGURED` | 设备自身状态：未进入可工作状态（永久，需用户配置） |
| 设备正在校准 | `CALIBRATING` | 设备自身状态：特殊过渡状态 |
| 前置命令未执行 | `DEPENDENCY_NOT_MET` | 设备自身状态：命令间的依赖关系未满足 |
| 安全/策略机制主动拒绝 | `SAFETY_INTERLOCK` | 设备自身策略：安全保护、儿童锁、环境保护等 |
| 物理障碍阻止执行 | `PHYSICALLY_OBSTRUCTED` | 外部物理世界：机械运动路径被阻挡 |
| 被并发操作抢占 | `CONCURRENT_MODIFICATION` | 并发来源：执行过程中状态被其他来源修改 |

## ⑥ 资源获取

前置条件满足，设备开始准备执行所需的资源。资源分为 **计算资源**（CPU、内存、队列、带宽）和 **物理资源**（耗材、子设备、外部依赖）两大类，穷尽了设备可能需要的一切：

| 失败原因 | 错误码 | 穷尽性说明 |
|---|---|---|
| CPU/执行单元被占用 | `DEVICE_BUSY` | 计算资源：单个操作占用 |
| 系统整体过载 | `DEVICE_OVERLOADED` | 计算资源：全局负荷过高 |
| 内存/存储/队列耗尽 | `RESOURCE_EXHAUSTED` | 计算资源：容量用尽 |
| 命令频率超限 | `RATE_LIMITED` | 计算资源：频率限制（保护硬件） |
| 子设备/被控设备不可达 | `CHILD_DEVICE_UNREACHABLE` | 物理资源：桥接器/网关无法联络下挂设备 |
| 耗材/消耗品耗尽 | `CONSUMABLE_DEPLETED` | 物理资源：执行依赖的消耗品用完 |

## ⑦ 物理执行

资源就绪，设备开始执行物理动作。执行阶段的失败只能来自 **执行器/硬件本身出问题**。按故障组件分类——硬件可分为**主控/通用硬件**、**传感器**、**电源**、**外设/子模块**四类（覆盖了设备的所有物理组成部分）；此外还有**软件层面**的执行失败，以及**执行超时**和**需要人工介入**：

| 失败原因 | 错误码 | 穷尽性说明 |
|---|---|---|
| 主控/通用硬件故障 | `HARDWARE_FAULT` | 硬件组件：主控、继电器、通信总线等 |
| 传感器故障 | `SENSOR_FAULT` | 硬件组件：传感器模块 |
| 电源问题 | `POWER_ISSUE` | 硬件组件：供电系统 |
| 外设/子模块故障 | `PERIPHERAL_FAULT` | 硬件组件：特定执行部件 |
| 固件/软件崩溃 | `SOFTWARE_FAULT` | 软件层：固件异常、看门狗重启 |
| 执行超时 | `DEVICE_TIMEOUT` | 时间维度：操作未在预期时间内完成 |
| 需人工物理介入 | `MANUAL_ACTION_REQUIRED` | 人工维度：设备检测到需人工操作才能继续（清理卡纸、清空水箱等） |

## 2.4 完备性结论

上述管线满足两个性质：

1. **阶段穷尽**：一条命令从到达设备到执行完成，必须且仅需通过①~⑦这七个阶段。命令不可能在管线之外的地方失败。
2. **阶段内穷尽**：每个阶段内部，失败原因按正交维度（值 vs 范围、单值 vs 多值、计算资源 vs 物理资源、硬件组件类型等）划分，确保不遗漏。

因此，本文档的错误码集合是 **完备的**——任何设备命令执行失败的场景都可以映射到恰好一个错误码。如果未来出现了无法映射的新场景，说明管线模型需要增加新阶段或新的正交维度，这要求对框架本身的修订（协议版本升级），而非简单地追加错误码。
