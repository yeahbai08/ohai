# OHAI 架构与安全设计

## OHAI 要解决的问题

智能家居设备的互联互通长期受制于各自为政的生态系统，用户被迫在多个 App 之间切换，无法实现真正的跨品牌自动化。虽然 Matter 协议试图通过统一标准解决这一问题，但其复杂的协议栈和高昂的认证门槛使得中小开发者和开源社区望而却步。

OHAI 的设计目标是：
1. 让消费者真正拥有对自己智能家居设备的完全控制权，打破生态壁垒，同时提供一个对 AI 友好的数据交互层，让大语言模型能够直接理解和操作设备状态。
2. 在协议层面实现智能家居设备的入网安全、传输安全以及大模型调用安全。

### 设备物理控制与 Server 离线

OHAI 设备**必须具备独立的物理控制能力**（物理开关、触摸面板等），设备核心功能不应完全依赖 Server 在线。当 Server 宕机或网络中断时，设备保持最后已知状态并继续响应物理操作。Server 恢复后设备自动重新连接，无需人工干预。

## 一、 设计哲学

OHAI (Open Home AI) 的 v2 设计旨在**借鉴 Matter 在安全方面的理念，同时降低复杂性**。

### 核心革新点
1. **Matter 级安全，极简体验**：引入 SPAKE2+ (PAKE) 协议用于设备配对，支持短数字配对码和动态 IP 网络。Console 与 Server 的信任建立通过公网认证服务 (TrustAnchor) 完成。
2. **公网辅助认证，本地自治运行**：初始设置借助 TrustAnchor 建立信任，日后日常运行完全在局域网内完成，无需持续联网。
3. **安全层与网络层解耦（类 Fabric 概念）**：更换 Wi-Fi 密码不再导致设备变砖重置。
4. **低门槛与开源根基**：不需要缴纳巨额联盟认证费，基于 mbedtls 即可在平价单片机（ESP32 等）上跑通。协议规范完全开放，第三方可自建 TrustAnchor。厂商通过 Open Vendor Registry (OVR) 以域名自证身份即可加入生态，无需审核或认证费用。

### 信任模型与安全边界

OHAI 协议中存在以下信任根，各自承担不同的职责：

| 信任根 | 职责范围 | 保障方式 |
|--------|---------|---------|
| **TrustAnchor** | Console ↔ Server 双向身份认证 | 证书签发 + JWT 签名 |
| **OVR** | 厂商身份验证、DAC Root CA 分发、Schema 公开托管 | 域名所有权自动验证 + append-only 审计日志（详见 [OVR 开放规范](#open-vendor-registry-ovr-开放规范)） |
| **设备厂商 PKI** | 设备硬件真实性 (DAC) | 厂商自建 |
| **设备 Secure Boot** | 固件完整性 | 逐级验签启动链 + 厂商签名密钥 HSM 管理 |
| **物理人工验证** | 设备配网授权 (SPAKE2+ Setup Code)、Server 绑定授权 (绑定码) | 用户物理接触设备/读取屏幕 |

**Console App 与 TrustAnchor 均可由第三方构建**，OHAI 协议不限制其实现者。协议保证的是：在用户选定的 TrustAnchor 和 Console App 均可信的前提下，上述信任链的安全性成立。至于具体选择哪个 TrustAnchor 实例、使用哪款 Console App，**其可信程度由用户自行判断**——正如用户选择浏览器或邮件客户端一样，协议定义交互规范，不为特定实现的可信度背书。

## 三、 身份与实体

### 1. 设备标识与配对凭证
* **Setup Code**: 11 位纯数字（如 `123-4567-8901`），采用 `XXX-XXXX-XXXX` 分组格式提高可读性。SPAKE2+ 的安全性主要依赖协议本身（而非密码熵），11 位（约 36.5 bit 熵）配合设备端的在线暴力破解防护（详见 Phase 1）已提供充足安全余量。相比 Matter 的 8 位配对码，额外的 3 位提供约 1000 倍的搜索空间作为安全冗余，而分组格式使手动输入的出错率保持在可接受范围内。二维码扫描为主要配对方式，手动输入仅作为后备。
  - **一次性失效**：Setup Code 在首次配对成功后，设备不再接受该码进行新的 SPAKE2+ 握手。攻击者即使事后获取包装盒上的二维码也无法重新配对。
  - **重新启用**：用户需对设备执行**物理出厂重置**（如长按重置按钮 10 秒）后，Setup Code 才重新生效，设备同时清除所有已存储的网络凭据和证书，回到初始可配对状态。
* **二维码载荷**: `ohai:V2?ID=xxx&C=12345678901&V=acme-iot.com`，其中 `V` 为厂商域名，用于 Console 预获取厂商 DAC Root CA。

### 2. 证书体系（借鉴 Matter 的 DAC 与 NOC）
* **DAC (Device Attestation Certificate)**: 设备出厂证书，证明其是”正品”设备。Server 在签发 NOC 前**必须验证**设备的 DAC 合法性——向设备发送随机 Challenge，要求设备用 DAC 私钥签名，再用 DAC 公钥验签，以此证明设备确实持有该证书对应的私钥。厂商 Root CA 公钥通过 Open Vendor Registry (OVR) 注册并公开分发，Server 定期从 OVR 同步到本地缓存，无需手动导入，也不受 Server 所选 TrustAnchor 的影响（详见 [OVR 开放规范](#open-vendor-registry-ovr-开放规范)）。
  - **私钥存储（强制要求）**：DAC 私钥**必须 (MUST)** 存储于设备内置的**安全元件 (Secure Element, SE)**（如 NXP SE050、Infineon OPTIGA Trust 等），私钥在 SE 内部生成且硬件级**不可导出 (non-extractable)**。所有涉及 DAC 私钥的签名操作均在 SE 内部完成，私钥明文永不暴露于主控 MCU 或外部总线。这是 DAC 防伪的根本保障——若私钥存储于普通 Flash，攻击者可通过拆机读取并批量克隆到假设备上，DAC 将完全丧失证明设备真实性的能力。
  - **证书签发**：厂商可根据自身能力选择不同的签发方式：
    - **自建 PKI**：厂商使用自签 Root CA 签发 DAC，设备出厂时将证书写入 SE。厂商将 Root CA 证书通过域名端点发布并在 OVR 注册，Server 自动同步获取。
* **NOC (Node Operational Certificate)**: 配网完成后，Server 为该设备签发的一张”本地家庭网证书”，该证书的 Root 是当前 Server。这一层完全隔绝了外部网络，建立了家庭内部的单一且安全的”信任织物 (Fabric)”。

### 3. 固件完整性 (Secure Boot)

DAC 证明的是设备硬件身份（”这颗芯片是正品”），而固件完整性需要另一道独立防线——**Secure Boot**（”运行的代码是正版”）。两者互补，缺一不可。

#### Secure Boot（强制要求）

OHAI 设备**必须 (MUST)** 实现完整的 Secure Boot 验证链：

- **逐级验签启动**：Boot ROM（芯片出厂固化，不可篡改）→ 验签 Second-stage Bootloader → 验签 Application。每一级在将控制权移交给下一级之前，必须验证其数字签名。任何一级验签失败，设备拒绝启动。
- **签名公钥固化**：厂商固件签名公钥（或其哈希）烧录于设备的 eFuse / OTP（一次性可编程）存储区，写入后不可更改。这确保攻击者无法通过替换公钥来使篡改固件通过验签。
- **参考实现**：ESP32 平台使用 Secure Boot v2（基于 ECDSA P-256 签名），参见 ESP-IDF 安全启动文档。

#### 固件签名密钥管理（强制要求）

固件签名密钥是 Secure Boot 信任链的根基，其安全性直接决定固件完整性保障的有效性：

- 签名私钥**必须 (MUST)** 存储于 HSM（硬件安全模块），不可导出。所有固件签名操作在 HSM 内部完成，私钥明文永不暴露于构建服务器或开发环境。
- **密钥轮换**：设备 eFuse 支持多签名密钥槽位时，厂商可通过 OTA 渐进部署新签名密钥——先用旧密钥签发包含新公钥的过渡固件，设备验证并烧录新公钥后，后续固件改用新密钥签名。
- **密钥泄露应急**：若签名密钥疑似泄露，厂商应立即发布安全公告，通过 OVR 发布安全通告标记受影响的设备批次（vendor_id + 批次范围），并尽快通过密钥轮换机制为在线设备部署新密钥。

#### 安全边界说明

Secure Boot 保证**只有经厂商私钥签名的固件才能在设备上运行**。未签名或被篡改的代码无法通过 Boot ROM 的验签，设备将拒绝启动。这是针对 ESP32 级硬件（无 TrustZone、无集成安全处理器）可实现的最高固件完整性保障。

### 4. 证书生命周期管理

#### NOC 有效期与自动续签
- NOC 有效期为 **1 年**。Server 在 NOC 到期前 30 天自动为设备签发新证书并通过 MQTT 下发，设备完成热切换，整个过程对用户透明。
- 若设备在证书到期前未完成续签（如长期离线），设备重新联网时 Server 自动检测并签发新 NOC。
- 若设备在 NOC 过期后才重新联网，Server 通过验证设备持有的原 NOC 私钥（Challenge-Response）确认身份后重新签发。

#### NOC 吊销
- Server 维护本地 **证书吊销列表 (CRL)**，当设备被移除时将其 NOC 加入 CRL。
- Server（作为 MQTT Broker）在 mTLS 握手阶段检查 CRL，拒绝已吊销设备的连接。
- CRL 仅存储于 Server 本地，无需外部分发。

#### Server Root CA 泄露恢复
- 若 Server Root CA 私钥疑似泄露，管理员通过 Console 触发 **Root CA 轮换**：
  1. Server 生成新的 Root CA 密钥对。
  2. Server 通过现有 MQTT 通道（旧 CA 仍在有效期内）向所有在线设备推送新 Root CA 公钥，并为每台设备重新签发 NOC。
  3. 设备完成热切换后，旧 Root CA 被标记为废弃。
  4. 离线设备重新联网时，Server 使用旧 CA 验证身份后完成迁移。
- Root CA 轮换期间系统保持运行，不中断设备控制。

## 四、 零配置安全入网流程 (Phase by Phase)

### Phase 0: Server 信任建立与 Console 绑定 (Bootstrapping)

OHAI Server 运行在家庭局域网内（如树莓派、NAS），无法获取公共 CA 签名的域名证书。为了让 Console App 能安全地验证 Server 身份，我们借助运行在公网上的 **TrustAnchor** 作为证书签发机构（CA），将 Server 身份纳入可信证书链。Console App 内置 TrustAnchor Root CA，从而通过标准 TLS 校验 Server 身份。

::: details 为什么需要 TrustAnchor，而非 Console 直接信任 Server？

绑定流程中用户已通过物理方式（肉眼读取绑定码）确认了 Server 的身份，一种看似更简单的替代方案是：Console 在首次绑定时直接 pin 住 Server 的公钥，后续连接验证 pinned key 即可，无需第三方参与。

这种**直接信任（Key Pinning）** 方案在最简场景（单 Console、单 Server、永不换手机）下确实可行，但在真实家庭使用的生命周期中会遇到以下问题，这正是引入 TrustAnchor 的根本原因：

1. **换手机 / 重装 App**：Pinned key 存储于手机本地，更换设备后新 Console 无法得知该信任哪个 Server 公钥。用户将被迫重新走一遍物理绑定码流程。而有了 TrustAnchor CA，新手机只需登录 OHAI 账户，App 内置的 CA 根证书即可通过标准 TLS 验证 Server 身份，零物理操作。

2. **灾难恢复（所有手机丢失）**：所有 pinned key 随手机丢失，用户必须物理接触 Server 主机才能重新建立信任。对于安装在柜内的 NAS 或树莓派，这极不便利。TrustAnchor 方案下，用户在新手机上登录账户即可恢复对 Server 的控制（详见下文「所有 Console 丢失的灾难恢复」）。

3. **密钥轮转**：安全最佳实践要求定期轮换身份密钥。Key Pinning 模型下密钥轮转会导致所有 Console 的 pinned key 失效，需要逐一重新物理验证。CA 模型下 Server 用新密钥申请新证书，Console 通过 CA 验证，对用户完全透明。

4. **多用户账户管理**：TrustAnchor 同时承担身份提供者角色，通过 JWT 为 Server 提供开箱即用的用户认证（账户 ID、角色权限、Token 过期与吊销）。若去掉 TrustAnchor，Server 需自行实现完整的用户认证系统，这对树莓派级硬件是额外的复杂度负担。

5. **标准化 TLS 校验**：Key Pinning 需要 Console 实现自定义的证书验证逻辑，而 CA 签发的证书可直接使用操作系统和 TLS 库的标准校验路径，降低实现出错的风险。

**取舍**：TrustAnchor 的代价是引入了一个公网依赖组件和额外的攻击面（若 TrustAnchor 被攻破，可签发伪造证书）。这一风险通过 HSM 强制要求和离线 Root CA 分层架构加以缓解（详见「TrustAnchor 开放规范」章节）。总体而言，Phase 0 中 TrustAnchor 的引入不是为了解决首次绑定的安全问题——物理验证已足够——而是为了支撑**绑定之后整个生命周期**的可用性与可管理性。

```text
       │                      │    → 匹配则签发证书     │
       │                      │    + Refresh Token   │
       │                      │ ────────────────────>│
       │                      │                      │
       │ ⑨ 重新 TLS 连接 Server (TrustAnchor CA 校验) │
       │ <═══════════════════════════════════════════│
       │                      │                      │
       │ ⑩ Console 出示 Access Token                  │
       │   (JWT: 账户ID + TrustAnchor 签名)       │
       │ ═══════════[TLS 通道]═════════════════════>│
       │                      │                      │
       │                      │  Server 验签令牌，     │
       │                      │  将该账户 ID 写入本地   │
       │                      │  授权账户列表 (管理员)  │
       │                      │                      │
       │ ✅ 双向信任建立：Console 验证 Server 身份，     │
       │    Server 确认 Console 账户授权              │
```

* **绑定码**：8 位纯数字（如 `1234-5678`），有效期 10 分钟，一次性使用。TrustAnchor 随机生成绑定码，并保证在所有未过期的活跃绑定码中唯一（生成时检查冲突，冲突则重新生成）。8 位数字提供 1 亿个可能值，在合理并发量下碰撞概率极低，同时保持用户手动输入的便利性。**暴力破解防护**：TrustAnchor 对绑定码验证接口施加严格速率限制——每个绑定码最多允许 5 次错误尝试，超出后立即作废；同时对每个账户/IP 施加滑动窗口限流（如每小时不超过 20 次验证请求），并在连续失败后引入指数退避延迟。
* **Console 触发绑定**：Server 启动后不会主动发送 CSR，而是等待 Console 通过临时 TLS 通道发出绑定请求后才发起。这解决了绑定码时效性问题——用户打开 Console App 这个动作本身就是"我准备好了"的信号，确保 10 分钟有效期内用户有充足时间完成操作。该临时通道未经认证，攻击者理论上可伪造请求触发 CSR 提交，但影响仅限于浪费一次 CSR 配额（受 IP 速率限制约束），绑定码只会显示在真实 Server 上，攻击者无法获取。
* **CSR 实时持有证明**：CSR 本身虽包含自签名（Proof of Possession），但无法防止攻击者获取他人 CSR 后原样重放。为此，TrustAnchor 在接收 CSR 后执行实时 Challenge-Response 验证：生成 32 字节随机 Challenge Nonce，要求提交方通过当前 HTTPS 会话用对应私钥签名该 Nonce 并回传。验签通过后才分配绑定码。这确保只有私钥持有者才能获得绑定码，即使 CSR 被截获也无法被重放利用。**超时与重放防护**：Challenge Nonce 有效期 60 秒且绑定当前 HTTPS 会话（Session-bound），过期或跨会话提交均视为无效；每个 Nonce 仅允许一次签名提交，防止重放。绑定码自分配起 10 分钟过期，到期自动作废，用户需在 Console 中重新触发绑定流程。
* **CSR 提交防滥用**：TrustAnchor 对 CSR 提交接口施加 IP 速率限制（如每 IP 每小时不超过 5 次）。由于公钥指纹双重校验已杜绝了误绑定风险，速率限制仅需防止 DoS 层面的资源耗尽。
* **证书签发时机**：Server 在 Console 触发下提交 CSR 并通过实时持有证明后，不会立即获得证书，TrustAnchor 仅在 Console 用户完成绑定码 + 公钥指纹校验后才签发。这确保每张证书都经过真人授权——防止攻击者自动获取合法证书。Server 在等待期间可继续完成其他本地初始化。
* **公钥指纹双重校验与安全模型**：TrustAnchor 以绑定码为路由键查找对应的 CSR，再验证 Console 提交的指纹与该 CSR 中的公钥是否一致，两者皆匹配后才签发证书。由于 TrustAnchor API 是公开的，安全性不依赖 Console App 可信——即使攻击者自建客户端直接调用 API，也必须同时提供正确的绑定码和指纹。这两个要素分别来自两条独立通道，构成双因素防护：
  - **绑定码（物理通道）**：仅显示在 Server 的物理屏幕或日志上，攻击者必须在场才能获取。远程暴力猜测受到严格速率限制（每 IP 每小时 20 次请求、每个绑定码 5 次错误即作废），在 10 分钟有效窗口内最多约 3 次尝试，命中 8 位绑定码的概率约为 3×10⁻⁸。
  - **公钥指纹（网络通道）**：Server 公钥的 SHA-256 摘要，攻击者必须在同一局域网内通过 TOFU（Trust On First Use）连接才能获取，远程无法猜测（256 位熵）。
  - **组合安全性**：不在局域网内的攻击者无法获取指纹，即使猜对绑定码也会因指纹不匹配被拒绝；在局域网内的攻击者可获取指纹，但仍需从 Server 物理屏幕获取绑定码，暴力猜测在速率限制下不可行。两条通道缺一不可，攻击者必须同时拥有物理临近和网络临近才能完成绑定——而满足这两个条件的人通常就是合法用户本人。
  - **误操作防护**：在正常使用中，合规的 Console App 在步骤③的 TOFU 连接中自动获取指纹，用户仅需手动输入绑定码。即使用户误输了其他 Server 的绑定码，指纹与该 CSR 的公钥不匹配，TrustAnchor 也会拒绝签发，防止跨 Server 误绑定。
* **Refresh Token**：TrustAnchor 签发给 Server 的长期凭证，用于后续自动证书续签（如每 90 天）。**存储安全**：Refresh Token 必须加密存储于 Server 本地，优先使用硬件安全模块（HSM/TPM）；若不具备硬件条件，则使用操作系统级密钥存储（如 Linux Kernel Keyring）或文件系统加密。Server 身份私钥同样遵循此存储规范。**Token 轮转**：每次证书续签成功时，TrustAnchor 同步签发新的 Refresh Token 并使旧 Token 失效（单次使用轮转），降低 Token 泄露后被长期利用的风险。
* **Console 连接授权**：TLS 证书仅验证 Server 身份（Console → Server 方向的信任），Server 还需反向验证 Console 的访问权限。Console 登录 OHAI 账户后从 TrustAnchor 获取 Access Token（JWT 格式，由 TrustAnchor 私钥签名），其中包含以下安全字段：`sub`（账户 ID）、`aud`（目标 Server 的公钥指纹，限定 Token 仅可用于特定 Server）、`exp`（过期时间，建议不超过 1 小时）、`jti`（唯一标识符，防重放）。TLS 握手成功后 Console 出示该令牌，Server 使用已有的 TrustAnchor CA 公钥验签，并校验 `aud` 是否匹配自身公钥指纹、`exp` 是否未过期、`jti` 是否已使用过（维护短期 jti 去重表），最后检查 `sub` 账户 ID 是否在本地**授权账户列表**中。验证全程在局域网内完成，无需联网。Phase 0 绑定时，第一个 Console 的账户 ID 自动写入授权列表并标记为管理员。
* **OHAI 账户**：每个家庭成员拥有独立的 OHAI 账户。管理员可通过 Server 邀请其他账户加入（详见下文「多 Console 管理」），也可单独吊销某个成员的访问权限，互不影响。
* **离线自治**：设置完成后，Server 仅在证书到期前联网续签，日常运行完全在局域网内完成，不依赖公网。
* **安全性**：Console App 仅信任 TrustAnchor CA 签发的证书，局域网内的中间人无法伪造合法的 Server 身份。绑定码 + 公钥指纹的双重校验确保只有局域网内物理可达的 Server 才能被绑定到用户账户。
* **mDNS 发现与 TOFU 安全边界**：步骤③中 Console 通过 mDNS 发现 Server 并建立未经认证的临时 TLS 连接（Trust On First Use），此连接**仅用于获取公钥指纹和触发绑定请求**，不传输任何敏感数据，不建立持久信任。即使局域网内存在 mDNS 欺骗攻击者，其影响范围被严格限制：攻击者可诱导 Console 获取错误指纹，但该指纹随后会与 TrustAnchor 上绑定码对应的 CSR 公钥进行交叉校验——指纹不匹配则拒绝签发证书，攻击者无法完成绑定。更进一步，即便用户同时输错了绑定码（恰好对应攻击者的 CSR），攻击者获得的证书也因缺少对应私钥而无法通过 TLS 握手（步骤⑨使用 TrustAnchor CA 严格校验）。

#### Commissioning Fail-Safe 机制

配网过程中设备可能因断电、BLE 断开、Console 崩溃等原因中途中断。若凭据写入不完整（例如已写入 Wi-Fi 密码但未写入 NOC，或 Setup Code 已失效但配网未完成），设备将陷入不可恢复的"半配网"状态。为此，OHAI 借鉴 Matter 的 Fail-Safe Timer 理念，引入 Commissioning Fail-Safe 机制：

**Fail-Safe 启动**：Console 在 SPAKE2+ 握手成功后（步骤②'）向设备发送 `ArmFailSafe(timeout)` 命令，设备启动 **Fail-Safe Timer**（默认 120 秒，Console 可在 30-600 秒范围内指定）。Fail-Safe 激活期间，所有凭据写入操作（Wi-Fi 密码、NOC、Recovery_PubKeys[]）写入设备的**暂存区 (staging area)**，不覆盖当前正式配置。

**提交 (Commit)**：Console 在所有凭据注入完成后发送 `CommissioningComplete` 命令。设备收到后执行以下原子操作：
1. 将暂存区凭据提交到正式存储区（NVS）
2. 将 Setup Code 标记为已用
3. 取消 Fail-Safe Timer
4. 关闭 BLE 广播，进入 Phase 2（Wi-Fi 连接 + mDNS 发现）

**回滚 (Rollback)**：以下任一情况触发自动回滚：
- Fail-Safe Timer 超时未收到 `CommissioningComplete`
- BLE 连接断开且未收到 `CommissioningComplete`
- Console 主动发送 `AbortCommissioning` 命令

回滚操作：
1. 清除暂存区所有已写入的数据
2. **Setup Code 保持有效**（未提交即不失效——这是关键设计点）
3. 设备返回初始可配对状态，继续 BLE 广播
4. SPAKE2+ 暴力破解失败计数器**不重置**（防止利用中断绕过速率限制）

**会话恢复**：若 Console 因 BLE 信号抖动短暂断开后迅速重连，可在 Fail-Safe Timer 未超时的情况下通过 Session_Key 验证身份一致性，继续之前的配网会话而无需从步骤①重新开始。这避免了因 BLE 不稳定导致频繁重试。

**实现要求**：
- 暂存区与正式存储区物理隔离（使用 NVS 的不同分区或命名空间），确保回滚时不会误清正式数据。
- Fail-Safe Timer 基于硬件定时器或 RTC 实现，不依赖 BLE 连接状态（即使 BLE 断开，Timer 仍在倒计时）。
- `CommissioningComplete` 命令必须在 Fail-Safe 激活状态下才被接受，否则返回错误。

* **DAC 前置验证**：Console 在步骤②之前先通过 BLE 读取设备的 DAC，并验证其签名链的合法性。DAC 验证失败的设备将被拒绝配对，Console 向用户提示"无法验证设备真实性"。
* **设备身份读取**：步骤②''中，Console 通过加密通道读取设备固件中固化的 `vendor`、`product`、`firmware_version` 三元组，转发给 Server 用于 Schema 查找（详见 [Schema 注册与认证](./secure-message-design.md#schema-注册与认证)）。此步骤在凭据注入（步骤③）之前完成，确保 Server 在配网结束前即完成 Schema 绑定。若 Server 在 OVR 中未找到对应 Schema，Console 通知用户进入开发者模式流程。
* **SPAKE2+ 在线暴力破解防护**：虽然 SPAKE2+ 协议本身能抵抗离线字典攻击，但仍需防护在线猜测。设备固件实现以下限制：
  - **尝试次数上限**：连续 10 次 SPAKE2+ 握手失败后，设备锁定 BLE 配对功能 30 分钟。
  - **指数退避**：第 3 次失败起引入延迟——第 3 次等待 2 秒，第 4 次 4 秒，第 5 次 8 秒……以此类推，第 10 次等待约 4 分钟。
  - **计数器持久化**：失败计数存储于非易失性存储（NVS），防止攻击者通过断电重置绕过。物理出厂重置会清除计数器。
* **Setup Code 一次性失效**：首次 SPAKE2+ 配对成功后，设备自动将 Setup Code 标记为已用，拒绝后续配对请求。需物理出厂重置后方可重新配对（详见第三节）。

## 五、 网络与信任的解耦 (抗“断网变砖”设计)

### 场景：家中更换路由器 / 修改 Wi-Fi 密码

OHAI 采用**分层恢复策略**——优先引导用户通过零成本方案恢复，逐级降级到 BLE 兜底。无论通过哪个层级恢复，全程不涉及安全重置或重复配对，设备与 Server 之间的信任关系始终有效。

#### 前置要求：Console App 存储 Wi-Fi 凭据

Phase 1 配网时 Console 已获取设备的 Wi-Fi SSID 与密码（步骤③）。Console App **必须 (MUST)** 将当前家庭网络的 Wi-Fi 凭据（SSID + Password）加密存储于操作系统安全沙盒（iOS Keychain / Android Keystore）。此凭据在后续恢复流程中用于：
- 提示用户复用旧 SSID/密码（层级 0）
- 引导用户临时恢复旧网络（层级 2）

凭据仅存储于 Console 本地，不上传至 TrustAnchor 或任何云端服务。

#### 智能断网检测

设备在 Wi-Fi 断连后，根据断连原因采取不同的等待策略，避免路由器正常重启时误触发恢复流程，同时在密码确实变更时快速响应：

| 断连原因 | 判断依据 | 等待策略 |
|---------|---------|---------|
| **AP 消失**（Wi-Fi Beacon 丢失） | 扫描不到原 SSID | 等待 **5 分钟**再进入 `Network_Lost`（路由器重启通常 1-3 分钟） |
| **认证失败**（密码错误） | 连接时收到 AUTH_FAIL | **30 秒**后尝试备用凭据（如有），**60 秒**后进入 `Network_Lost` |
| **DHCP 失败 / IP 冲突** | 关联成功但无法获取 IP | 等待 **3 分钟**（给路由器 DHCP 服务恢复时间） |

- 用户可通过**2 秒内快速双击重置按钮**（设备 LED 慢闪确认）立即进入 `Network_Lost` 状态，无需等待超时。
- 进入 `Network_Lost` 状态后，设备在后台持续尝试重新连接原 Wi-Fi 及备用凭据（每 30 秒轮询一次）。若任一 Wi-Fi 重连成功，设备立即关闭 BLE 广播、恢复 MQTT 连接并回到正常运行状态，无需人工干预。

#### 恢复层级 0：SSID 与密码复用（零成本，推荐首选）

当 Console 检测到设备批量离线时，首先引导用户采用最简单的恢复方式——**将新路由器的 Wi-Fi 名称和密码设置为与旧路由器相同**：

- Console App 弹出恢复引导，展示存储的旧 Wi-Fi 凭据：*”您的 {N} 台设备配置在 Wi-Fi「{stored_ssid}」。将新路由器的 Wi-Fi 名称和密码设为相同即可自动恢复，无需逐台操作。”*
- 用户在新路由器管理界面中修改 SSID 和密码后，设备在下一次重试周期（≤ 30 秒）内自动重连。
- **适用场景**：更换路由器（旧 SSID/密码可查）。这是现实中最常见的更换路由器方式。
- **不适用**：用户刻意要换新密码（如旧密码泄露）、ISP 设备不允许自定义 SSID、Console 未存储旧凭据。

#### 恢复层级 1：计划性迁移（通过 MQTT 预置凭据）

当用户需要使用全新的 SSID 或密码时，Console App 提供**”计划更换 Wi-Fi”向导**，在改路由器之前将新凭据预置到所有设备：

```text
┌──────────────┐                          ┌──────────────┐       ┌──────────────┐
│   Console    │                          │    Server    │       │   Devices    │
│ (Mobile App) │                          │   (中枢)     │       │  (所有设备)   │
└──────┬───────┘                          └──────┬───────┘       └──────┬───────┘
       │                                         │                      │
       │ ① 用户在向导中输入新 SSID + Password     │                      │
       │ ═══════════[TLS 通道]══════════════════>│                      │
       │                                         │                      │
       │                ② Server 通过 MQTT 加密推送                      │
       │                  备用凭据至所有在线设备                           │
       │                                         │ ────────────────────>│
       │                                         │                      │
       │                                         │  ③ 设备确认已收到     │
       │                                         │ <────────────────────│
       │                                         │                      │
       │ ④ Console 显示: “所有设备已准备就绪，      │                      │
       │   请前往路由器管理界面更改 Wi-Fi 设置。”    │                      │
       │                                         │                      │
       │            ⑤ 用户更改路由器 Wi-Fi 设置                           │
       │                                         │                      │
       │                                         │  ⑥ 设备断连后自动     │
       │                                         │    尝试备用凭据       │
       │                                         │    连接成功 → 提升    │
       │                                         │    为正式凭据         │
       │                                         │                      │
```

- 设备将备用凭据存储于独立的 **备用凭据存储区 (Standby Credentials)**，与正式凭据隔离。
- Console 等待所有在线设备确认收到后（MQTT QoS 1 + 设备端 ACK），才提示用户去改路由器。对于离线设备，Console 明确提示哪些设备未收到预置，这些设备将在后续通过层级 3 恢复。
- **适用场景**：用户主动计划更换 Wi-Fi，且通过 Console App 发起变更。
- **不适用**：用户已经直接在路由器上改了 Wi-Fi（事后才打开 App）。

#### 恢复层级 2：临时恢复旧网络 + 预置

当用户已经更改了 Wi-Fi 导致设备离线，但旧密码可查（Console 本地存储 / 用户记得）时，Console 引导用户**临时恢复旧网络**以触发自动预置：

1. Console 检测到设备批量离线，且层级 0 不适用（用户明确要用新密码）。
2. Console 引导：*”请暂时将路由器的 Wi-Fi 密码改回「{old_password}」。设备重连后，App 将自动预置新密码，届时再切换即可。”*
3. 用户将路由器密码改回旧密码 → 设备在 ≤ 30 秒内自动重连。
4. Console 检测到设备全部上线后，自动执行层级 1 的预置流程（推送新凭据至所有设备）。
5. 全部设备确认收到后，Console 提示：*”所有设备已准备就绪，请将路由器密码更改为您的新密码。”*
6. 用户将路由器密码改为目标新密码 → 设备自动切换至备用凭据。

- **适用场景**：同一路由器修改密码（旧密码可查，路由器仍在）。
- **不适用**：旧路由器已不存在、旧密码不可查。
- **诚实说明**：此流程需要用户修改两次路由器密码，体验不如层级 0/1，但优于逐台 BLE 恢复。Console App 应将全流程步骤化呈现，引导用户逐步操作，降低认知负担。

#### 恢复层级 3：BLE Recovery（兜底方案）

当上述层级均不可行时（旧路由器已不存在、旧密码不可查、未提前预置），设备只能通过 BLE 恢复。这是成本最高的恢复路径，但确保在任何情况下设备都不会”变砖”。

**退避降级**：设备进入 `Network_Lost` 状态后自动唤醒 BLE 广播（详见上文「智能断网检测」）。

**BLE 自动化轮询与静默认证**：
   - 现代智能手机的 BLE 通常不支持同时与几十个设备建立并发连接。
   - **解法**：Console (App) 扫描并收集所有处于 `Network_Lost` 状态的设备广播，随后发起**自动化轮询**（连上一个 → 验证发密码 → 断开 → 连下一个）。
   - **静默认证机制 (关键点)**：
     * **配对时留存**：在 Phase 1 首次配网通过 SPAKE2+ 建立信任通道时，App (Console) 会生成一对 ECDSA P-256 密钥，并将公钥写入设备安全存储区内的 **恢复密钥列表 (`Recovery_PubKeys[]`)**。App 自身则在操作系统的安全沙盒（iOS Keychain / Android Keystore）中保管对应的私钥。后续通过 Server 授权的其他 Console 的公钥也会被追加到此列表中（详见下文「多 Console 管理」）。
     * **恢复时验证 (Authenticated Key Exchange)**：当设备断网广播 `Network_Lost` 时，它**不再接受**新的 Setup Code 配网，而是只接受 `Recovery_PubKeys[]` 列表中任意一个私钥持有者。Console 连接设备后，双方执行以下认证密钥交换协议，同时完成身份验证和前向安全的会话密钥协商：
       1. **设备 → Console**：设备生成 32 字节随机 Challenge Nonce 和临时 ECDH P-256 密钥对（`Eph_PK_device`），发送 `{Challenge, Eph_PK_device}`。
       2. **Console → 设备**：Console 生成自己的临时 ECDH P-256 密钥对（`Eph_PK_console`），使用 Recovery 私钥签名：`sig = ECDSA_Sign(Recovery_SK, Challenge ‖ Eph_PK_device ‖ Eph_PK_console)`，发送 `{sig, Eph_PK_console, Recovery_PK_Index}`。
       3. **设备验证**：设备根据 `Recovery_PK_Index` 查找对应公钥（不匹配则遍历列表，列表通常不超过 5 条目），验证 ECDSA 签名。签名覆盖了 Challenge（防重放）和双方临时公钥（防中间人篡改）。
       4. **会话密钥派生**：双方各自计算 `shared_secret = ECDH(Eph_SK_self, Eph_PK_peer)`，再通过 HKDF-SHA256 派生会话密钥：`Session_Key = HKDF(shared_secret, salt=Challenge, info=”ohai-recovery”)`。后续通信使用 AES-256-GCM 加密。
       5. **前向安全性**：临时 ECDH 密钥在会话结束后立即销毁。即使 Recovery 私钥日后泄露，攻击者也无法解密历史恢复会话中传输的 Wi-Fi 凭据。
     * **快速接管**：上述认证密钥交换仅需一个完整 BLE 往返（两条消息），设备端 ECDSA 验签 + ECDH 计算在 ESP32 级硬件上耗时约 100-200 毫秒。会话密钥建立后，新的 Wi-Fi 凭据经 AES-256-GCM 加密下发，设备解密写入后自动尝试连接新 Wi-Fi。
     * **恢复尝试速率限制**：虽然攻击者伪造 ECDSA P-256 签名在计算上不可行，设备仍对 BLE Recovery 连接施加纵深防御——连续 5 次签名验证失败后锁定 BLE Recovery 接口 5 分钟。失败计数持久化于 NVS，断电不重置；物理出厂重置可解除锁定。
   - **Wi-Fi 在线设备优先更新**：对于更换路由器但新旧 Wi-Fi 共存的过渡期，仍可连接旧 Wi-Fi 的设备可通过 Server MQTT 通道直接接收新 Wi-Fi 凭据并自动切换，无需 BLE 恢复。仅完全断网的设备才需要 BLE 轮询。
   - **BLE 恢复的现实约束**：BLE 为串行连接，每台设备约需 3-5 秒（含连接、认证、下发、断开），30 台设备预计耗时 2-3 分钟。BLE 有效距离通常为 10-15 米，穿墙会显著衰减，用户可能需要在不同房间分别操作。Console App 应显示实时进度并标注尚未恢复的设备，帮助用户定位需要靠近的区域。

#### Wi-Fi 凭据更新 Fail-Safe 机制

无论通过上述哪个层级下发新的 Wi-Fi 凭据，设备均执行以下原子性更新流程，确保不会因凭据错误导致设备卡死：

1. 新凭据写入**暂存区 (Staging)**，不覆盖当前正式凭据。
2. 设备断开当前 Wi-Fi（如仍连接），尝试使用暂存区凭据连接新 Wi-Fi，设置 **60 秒超时**。
3. **连接成功**：原子提交——暂存区凭据提升为正式凭据，原正式凭据降级为 **fallback 凭据**（保留一份，以备回退）。设备通过新 Wi-Fi 恢复 MQTT 连接，向 Server 上报凭据切换成功。
4. **连接失败**（认证失败 / AP 不可达 / DHCP 超时）：回滚——清除暂存区，恢复使用原正式凭据尝试重连。若原凭据也无法连接，设备进入 `Network_Lost` 状态。设备在下次 BLE Recovery 或 MQTT 连接时向 Console 报告失败原因。

#### 恢复效果

| 恢复层级 | 用户操作 | 恢复耗时 | 适用前提 |
|---------|---------|---------|---------|
| **层级 0** | 在新路由器上设置旧 SSID/密码 | 设备 ≤ 30 秒自动重连 | 旧凭据可查 |
| **层级 1** | 在 Console 中预置新凭据，再改路由器 | 设备 ≤ 30 秒自动切换 | 用户通过 App 主动发起 |
| **层级 2** | 临时改回旧密码，等自动预置后改新密码 | 约 5-10 分钟 | 旧密码可查，路由器仍在 |
| **层级 3** | 手机靠近设备，App 自动 BLE 轮询 | 5-20 分钟（取决于设备数量和物理距离） | 任何情况（兜底） |

所有层级的共同保证：**不涉及安全重置，不需要重新配对，设备与 Server 之间的信任关系始终有效。**

### Network_Lost BLE 广播隐私保护

设备在 `Network_Lost` 状态下通过 BLE 广播自身身份，可能被攻击者利用来追踪设备位置或推断家庭设备组成。为此，OHAI 定义以下隐私保护措施：

- **BLE 私有地址轮换**：设备在 `Network_Lost` 状态下使用 **BLE Resolvable Private Address (RPA)**，基于设备内置的 IRK (Identity Resolving Key) 每 15 分钟轮换一次 MAC 地址。已授权的 Console 在首次配网时获取设备 IRK（通过 SPAKE2+ 加密通道传输），可解析 RPA 识别设备；未授权方仅看到不断变化的随机地址，无法关联到特定设备。
- **广播载荷最小化**：BLE 广播的 Service Data 字段仅包含以下最小信息：
  - OHAI Service UUID（标识协议类型）
  - 4 字节**设备标识提示 (Device Hint)**：`HMAC-SHA256(Recovery_PK[0], device_id)[0:4]`
  - 1 字节状态标志（`Network_Lost` 标志位 + 协议版本）

  Console 使用本地存储的设备列表计算 HMAC 比对，即可快速定位自己管理的设备。攻击者即使长期监听，也无法从 4 字节截断 HMAC 还原 `device_id`，且因 RPA 轮换无法跨时间段关联同一设备的广播。
- **广播频率退避**：进入 `Network_Lost` 后，BLE 广播间隔随时间递减以节省电量并减少暴露面：
  - 前 10 分钟：100ms 间隔（快速被发现）
  - 10 分钟 ~ 24 小时：1 秒间隔
  - 24 小时后：10 秒间隔（长期待恢复模式）

  用户通过物理双击重置按钮可使设备重新进入 100ms 快速广播阶段（持续 10 分钟）。

### Thread/802.15.4 Mesh 设备支持

WiFi 设备在网络凭据变更时依赖上述 BLE 恢复机制，而低功耗电池设备（门窗传感器、温湿度传感器、人体感应器等）通常无法负担 WiFi 的功耗。OHAI 通过集成 **Thread（IEEE 802.15.4）** mesh 网络为此类设备提供支持。

**Thread 的核心优势在于网络韧性**：Thread 网络运行在独立的 802.15.4 射频层上，与 WiFi 完全隔离。**WiFi 密码变更仅影响 Thread Border Router 的上行 WiFi 连接，Thread 终端设备不受任何影响。** 这与 Matter 中 Thread 设备天然免疫 WiFi 变更的原理一致。

#### Thread Border Router (TBR) 集成

OHAI Server 可通过外接 802.15.4 射频模块（如 nRF52840 USB Dongle、Silicon Labs EFR32 模组）集成 Thread Border Router 功能，桥接 Thread 网络与 IP 网络：

```text
                    ┌─────────────┐
                    │ OHAI Server │
                    │ ┌─────────┐ │
 WiFi               │ │  TBR    │ │               802.15.4
◄──────────────────►│ │ (内置)  │ │◄───────────► Thread Mesh
 Console / WiFi Dev │ │ MQTT-SN │ │               低功耗设备
                    │ │ Gateway │ │
                    │ └─────────┘ │
                    └─────────────┘
                          ▲
                          │ Thread Mesh
            ┌─────────────┼─────────────┐
            │             │             │
     ┌──────┴──────┐ ┌───┴────┐ ┌──────┴──────┐
     │ 门窗传感器   │ │温湿度   │ │  人体感应    │
     │ (SED)       │ │(SED)   │ │  (SED)      │
     └─────────────┘ └────────┘ └─────────────┘
```

- **TBR 也可作为独立设备**：若 Server 硬件不具备 802.15.4 射频，TBR 可作为独立硬件存在（如专用 Thread Border Router），通过 WiFi 连接 Server。在此架构下，WiFi 密码变更时仅需恢复 TBR 的 WiFi 连接（走标准 BLE Recovery），Thread 终端设备完全不受影响。
- **多 TBR 冗余**：Thread 规范支持同一网络内多个 Border Router 共存。当一个 TBR 离线时，Thread 设备可通过其他 TBR 路由到 Server，提供自愈能力。OHAI 建议具备条件的家庭部署至少两个 TBR。

#### Thread 设备配网

Thread 设备的配网**复用 Phase 1 的 BLE + SPAKE2+ 流程**，仅在凭据注入内容上有所不同：

| | WiFi 设备 | Thread 设备 |
|---|---|---|
| BLE 配对 | SPAKE2+（相同） | SPAKE2+（相同） |
| 注入凭据 | WiFi SSID + Password | Thread Network Key + PAN ID + Channel + Extended PAN ID + Mesh-Local Prefix |
| Server 证书链 | 相同 | 相同 |
| NOC 签发 | 相同 | 相同 |
| Recovery_PubKeys[] | 相同 | 相同（用于 Thread 凭据更新） |

配网完成后，Thread 设备使用注入的 Thread 网络凭据加入 mesh。设备通过 Thread 网络中的 mDNS-over-SRP（Service Registration Protocol）发现 Server 的 TBR 地址，随后建立安全连接。

#### Thread 设备通信协议

Thread 设备资源受限（典型 RAM < 64 KB），无法运行完整的 MQTT + mTLS 协议栈。OHAI 采用 **MQTT-SN (MQTT for Sensor Networks)** 作为 Thread 设备的应用层协议：

- **MQTT-SN over UDP/DTLS 1.3**：Thread 设备使用 MQTT-SN 协议通过 UDP 通信，TBR 内置 MQTT-SN Gateway 将其透明转换为标准 MQTT 消息。传输层使用 **DTLS 1.3** 替代 mTLS，设备使用 NOC 进行双向认证，安全等级与 WiFi 设备的 mTLS 等效。
- **Sleepy End Device (SED) 支持**：电池供电的 Thread 设备大部分时间处于休眠状态，定期唤醒（可配置，典型值每 30 秒 ~ 5 分钟）向父路由器拉取缓存消息。MQTT-SN 的 `REGISTER` + `PUBLISH` 机制天然兼容此模式——SED 唤醒后发布传感器数据，拉取待下发命令，然后迅速回到休眠。Server 通过 MQTT-SN Gateway 为 SED 缓存命令，确保命令在设备下次唤醒时送达。
- **Topic 映射**：MQTT-SN 使用短整数 Topic ID（2 字节）替代完整 Topic 字符串，显著节省受限网络的带宽。TBR 内的 MQTT-SN Gateway 维护 Topic ID ↔ MQTT Topic 的映射表，对 Server 侧完全透明——Server 统一使用标准 MQTT Topic，无需感知底层是 WiFi 设备还是 Thread 设备。

#### Thread 网络韧性

| 故障场景 | 影响 | 恢复方式 |
|---|---|---|
| **WiFi 密码变更** | Thread 终端设备**不受影响**；TBR（如果通过 WiFi 上行）需恢复 WiFi 连接 | TBR 作为 WiFi 设备走标准 BLE Recovery；Thread mesh 在 TBR 恢复后自动重建路由 |
| **TBR 离线** | 若仅有一个 TBR，Thread 设备暂时无法与 Server 通信（但 mesh 内部设备间通信正常） | 多 TBR 冗余自动故障切换；或等待 TBR 恢复后自动重连 |
| **Thread Network Key 泄露** | Thread 网络层加密被突破 | Server 通过 TBR 向所有在线 Thread 设备推送新的 Network Key（Thread 1.3.0+ 支持 Key Rotation）；长期离线设备通过 BLE Recovery 机制注入新 Thread 凭据 |
| **Thread 设备固件 OTA** | 受限设备带宽低，更新耗时长 | 通过 Thread 网络分片传输（MQTT-SN 分片 + 应用层重组），TBR 缓存固件分片，SED 在每次唤醒时拉取下一分片 |

### 多 Console 管理与密钥生命周期

上述恢复机制依赖 `Recovery_PubKeys[]` 列表，以下定义该列表的完整管理流程，确保不会因单台手机丢失而导致所有设备不可恢复。

#### 1. 添加新 Console（家庭成员加入）

#### 2. Console 吊销（手机丢失 / 成员退出）

```text
┌──────────────┐                          ┌──────────────┐
│ Admin Console│                          │    Server    │
│  (管理员手机) │                          │   (中枢)     │
└──────┬───────┘                          └──────┬───────┘
       │                                         │
       │ ① 管理员在 App 中将目标成员标记为吊销       │
       │ ═══════════[TLS 通道]══════════════════>│
       │                                         │
       │                 ② Server 从授权账户列表移除该账户 ID
       │                   同时从注册表移除其恢复公钥
       │                   通过 MQTT 向所有在线设备推送
       │                   更新后的 Recovery_PubKeys[]
       │                                         │
       │ ✅ 被吊销成员立即丧失连接与恢复能力          │
```

* **离线设备的吊销同步**：与添加流程相同，设备重新联网时 Server 补推。在此之前，若被吊销的 Console 试图通过 BLE 恢复离线设备，可由以下机制缓解：
  - Server 在吊销时生成一个带时间戳的**吊销令牌 (Revocation Token)**，由 Server 私钥签名。
  - 其他已授权 Console 在 BLE 恢复时可携带此令牌，设备验签后即时剔除对应公钥。

#### 3. 所有 Console 丢失的灾难恢复

当所有手机均丢失时，用户仍可通过 OHAI 账户恢复控制：

1. 在新手机上安装 Console App，登录 OHAI 账户（通过账户密码恢复流程）。
2. Console 通过 TrustAnchor CA 校验并连接 Server（无需物理访问 Server 主机）。
3. 新 Console 注册恢复公钥后，Server 向所有在线设备推送新的 `Recovery_PubKeys[]`。
4. 对于已断网的设备，新 Console 通过 BLE 恢复时需配合 Server 签发的**紧急恢复令牌 (Emergency Recovery Token)**——由 Server 私钥签名、包含新 Console 公钥的一次性凭证。设备使用已存储的 Server Root CA 公钥验证该令牌的合法性，验证通过后将新公钥加入 `Recovery_PubKeys[]` 并替换旧列表。

> **安全约束**：紧急恢复令牌仅在所有已注册 Console 均被吊销后才可签发，且有效期不超过 24 小时，防止被滥用为常规配网手段。

::: tip 用户操作指南：所有手机丢失后如何恢复
1. **新手机装 App**：在新手机上安装 OHAI Console App。
2. **登录账户**：使用你的 OHAI 账户邮箱和密码登录（如忘记密码，通过邮箱重置）。
3. **自动恢复在线设备**：登录后 App 会自动发现并连接家中的 Server，所有在线设备将自动恢复控制，无需额外操作。
4. **恢复离线设备**：对于已断网的设备，App 会引导你逐一靠近设备，通过蓝牙自动完成身份验证和恢复，全程按屏幕提示操作即可。
:::

#### 4. 用户权限分级

Server 的授权账户列表中，每个账户关联一个**角色 (Role)**，决定其可执行的操作范围：

| 角色 | 设备控制 | 配网新设备 | 邀请/吊销成员 | 修改 Server 设置 | 恢复密钥 |
|------|---------|-----------|-------------|----------------|---------|
| **管理员 (Admin)** | ✅ | ✅ | ✅ | ✅ | ✅ 写入设备 |
| **成员 (Member)** | ✅ | ✅ | ❌ | ❌ | ✅ 写入设备 |
| **访客 (Guest)** | ✅ 受限 | ❌ | ❌ | ❌ | ❌ |

- **管理员**：Phase 0 绑定时第一个 Console 账户自动成为管理员，拥有完整权限。
- **成员**：家庭常驻成员，可控制设备和配网新设备，但不能管理其他账户或修改 Server 配置。
- **访客**：临时访问权限，由管理员/成员授予。访客的 Access Token 附带额外约束：`exp` 有效期更短（建议不超过 24 小时），且可限定可控制的设备范围（通过 Token 中的 `scope` 字段列出允许的设备 ID 列表）。过期后自动失效，无需手动吊销。访客不持有恢复密钥，无法通过 BLE 恢复设备。

## 六（附）、 固件 OTA 更新安全

设备固件更新是 IoT 安全的关键一环。OHAI v2 定义以下 OTA 安全机制：

### 签名验证
- 固件包必须由厂商使用 **ECDSA P-256** 进行数字签名，与 Secure Boot 使用同一签名体系和同一把固件签名密钥（详见第三节「固件完整性」）。这确保 OTA 写入的固件在下次启动时能通过 Secure Boot 验证——若 OTA 固件未经正确签名，即使写入成功，设备重启后 Secure Boot 也会拒绝启动，自动回滚到上一个有效固件分区（A/B 分区方案）。
- 设备出厂时将厂商固件签名公钥（或其哈希）烧录于 eFuse/OTP，OTA 时先验签再刷写，验签失败则拒绝更新。

### 分发通道
- Server 可作为 OTA 代理：厂商将签名固件包上传至公开的更新服务器，Server 定期检查并下载后通过 MQTT 通知设备拉取（或直接推送分片）。设备无需直接访问公网。
- 固件元数据（版本号、SHA-256 哈希、签名）通过 MQTT `ohai/{device_id}/ota` topic 下发，设备校验后通过 HTTP 从 Server 局域网地址拉取固件包。

### 防降级 (Anti-Rollback)
- 设备在安全存储区维护一个**单调递增的版本计数器 (Monotonic Counter)**，每次成功刷写后递增。固件包中声明的最低版本计数器低于当前值时拒绝刷写，防止攻击者回滚到存在已知漏洞的旧版本。

## 六（附）、 设备移除、转让与出厂重置

### 1. App 端设备移除

管理员/成员通过 Console App 发起设备移除：
1. Console 向 Server 发送移除指令。
2. Server 将该设备的 NOC 加入本地 CRL。
3. Server 通过 MQTT 向设备发送 `factory_reset` 命令。
4. 设备收到命令后清除所有存储的凭据（NOC、Wi-Fi 密码、Recovery_PubKeys[]），恢复到出厂可配对状态，Setup Code 重新生效。

### 2. 物理出厂重置

用户可在无网络的情况下通过**物理操作**重置设备（如长按重置按钮 10 秒，设备 LED 快闪确认）：
- 清除所有网络凭据、证书和恢复密钥。
- Setup Code 重新生效，设备进入 BLE 广播可配对状态。
- 重置不可撤销，设备与原 Server 的信任关系完全断开。
- Server 侧仍需管理员手动移除该设备记录（或在设备长期离线后自动标记为"已脱离"）。

### 3. 设备转让

将设备转让给其他家庭时：
1. 原管理员在 Console 中移除设备（步骤同上）。
2. 设备回到出厂状态后，新家庭按正常 Phase 1 流程配网即可。
3. 无需厂商介入，无需联网操作。

## TrustAnchor 开放规范

### 定位

TrustAnchor 是 OHAI 协议中的公网认证服务，专职为家庭内网的 OHAI Server 签发可信身份证书并管理用户账户，使 Console App 能通过标准 TLS 校验 Server 身份。厂商信任（DAC Root CA 分发与 Schema 托管）由独立的 Open Vendor Registry (OVR) 负责（详见 [OVR 开放规范](#open-vendor-registry-ovr-开放规范)），TrustAnchor 不涉及厂商或设备相关数据。

### 开放性原则

- **API 规范完全开放**：OHAI 协议规范定义了 TrustAnchor 的完整 API 接口标准（CSR 提交、绑定码验证、证书签发、证书续签），任何组织或个人均可按照规范自建 TrustAnchor 实例。
- **官方默认实例**：OHAI 官方运营一个默认的 TrustAnchor 服务，官方 Console App 默认连接此实例。
- **可配置端点**：OHAI Server 的 TrustAnchor 地址为可配置项，支持指向第三方自建实例。第三方 Console App 可选择连接任意兼容的 TrustAnchor。

### TrustAnchor 仅存储的数据

| 数据 | 用途 |
|------|------|
| 用户账户信息 (邮箱/密码哈希) | 账户认证 |
| Server 公钥指纹 + 证书 | 身份签发与续签 |
| Server 与账户的绑定关系 | 归属标识 |

TrustAnchor **不存储也不接触**：设备信息、厂商信息、设备 Schema、家庭网络拓扑、控制指令、用户使用数据。厂商相关数据由 OVR 独立管理。

### 证书续签

Server 身份证书有效期为 90 天。到期前 Server 使用 Refresh Token 自动向 TrustAnchor 申请续签，整个过程无需用户干预。若 Server 长期离线导致证书过期，用户重新在 Console App 中触发一次绑定码流程即可恢复。

**续签失败处理**：Server 应在证书到期前 30 天开始尝试续签，每次失败后以指数退避重试（初始间隔 1 小时，最大间隔 24 小时）。证书到期前 7 天若仍未续签成功，Server 通过 MQTT 向已连接的 Console 推送告警通知。证书过期后，Server 停止接受新的 TLS 连接，但已建立的 MQTT 会话保持活跃直至自然断开，确保不会因续签失败立即中断设备控制。

### 证书吊销

当 Server 被确认遭到入侵或所有者需要解除绑定时，管理员可通过 Console App 向 TrustAnchor 发起证书吊销请求。TrustAnchor 维护一份轻量级**证书吊销列表 (CRL)**，Console App 在每次 TLS 连接 Server 前检查 CRL（缓存有效期 24 小时，无法联网时使用缓存）。由于 OHAI 证书有效期仅 90 天（远短于传统 CA 的 1-2 年），即使 CRL 传播存在延迟，风险窗口也被控制在较短时间内。

### 密钥安全要求

TrustAnchor 持有两类核心私钥：**CA 签名私钥**（签发 Server 身份证书）和 **JWT 签名私钥**（签发 Console Access Token）。这两个私钥是整个 OHAI 信任体系的根基，必须执行最高级别的保护。

#### 硬件安全模块 (HSM)
- CA 签名私钥和 JWT 签名私钥**必须**存储于硬件安全模块 (HSM) 中，私钥在 HSM 内生成且**不可导出 (non-extractable)**。
- 所有签名操作（证书签发、JWT 签名）通过 PKCS#11 或 KMS API 在 HSM 内部完成，私钥明文永不暴露于主机内存。

#### 离线 Root CA 分层架构（推荐）
- 推荐采用**离线 Root CA + 在线 Intermediate CA** 的分层架构：
  - **Root CA 私钥完全离线**，存储于物理隔离的 HSM 中，仅在签发或轮换 Intermediate CA 证书时启用。
  - **Intermediate CA 在线运行**，通过 HSM 保管私钥，负责日常的 Server 证书签发。
  - 即使在线 Intermediate CA 被攻破，Root CA 仍然安全，可签发新的 Intermediate CA 并吊销旧的。
- Console App 内置 Root CA 证书，信任链不变。

#### 密钥仪式 (Key Ceremony)
- Root CA 私钥的生成必须在**气隙隔离 (air-gapped)** 环境中进行，多人在场见证并记录。
- HSM 激活凭证采用 **Shamir's Secret Sharing** 分片交给多位密钥持有人（M-of-N 门限方案），任何签名操作需至少 M 位持有人同时在场。

#### 运维安全
- **最小权限**：仅极少数经过背景审查的运维人员有权访问 HSM。
- **审计日志**：所有签名操作记录至不可篡改的审计日志，异常签发量触发即时告警。
- **密钥轮换**：JWT 签名密钥定期轮换，通过 JWKS (JSON Web Key Set) 端点发布当前有效公钥集，Server 缓存并定期刷新。
- **灾备**：HSM 密钥备份至异地 HSM（加密传输），防止单点故障。

#### 自建实例的安全基线
第三方自建 TrustAnchor 实现者**应当遵循相同的密钥保护标准**。协议规范将 HSM 存储列为强制要求（MUST），离线 Root CA 分层架构列为推荐要求（SHOULD）。不满足 HSM 要求的实现不应用于生产环境。

## Open Vendor Registry (OVR) 开放规范

### 定位

OVR（开放厂商注册表）是独立于 TrustAnchor 的**全局共享**基础设施，负责厂商 DAC Root CA 公钥分发和设备 Schema 定义的公开托管。

TrustAnchor 的职责是为 Server 签发身份证书和管理用户账户——这是**局部的**，每个家庭自选 TrustAnchor，不同家庭可以使用不同的 TA 实例。而厂商 DAC Root CA 和设备 Schema 是**全局的**——所有用户的 Server 都需要能验证所有厂商的设备。若将全局资源绑定在局部服务上，使用不同 TrustAnchor 的用户将无法验证对方生态中厂商的设备，导致 OHAI 碎片化为多个互不兼容的孤岛，违背"打破生态壁垒"的初衷。因此，OVR 必须独立于 TrustAnchor 存在。

### 域名锚定身份 (Domain-Anchored Vendor Identity)

OVR 采用**域名所有权**作为厂商身份的信任锚点，通过自动化协议验证，**无需人工审核**。这与 Let's Encrypt 签发 TLS 证书的模式一致——Let's Encrypt 每年签发数亿张证书，零人工审核，靠的就是自动化域名验证。

#### 厂商自证端点

每个厂商在其域名下发布标准化的厂商信息文件：

```json
// https://acme-iot.com/.well-known/ohai/vendor.json
{
  "vendor_id": "acme-iot",
  "vendor_domain": "acme-iot.com",
  "display_name": "Acme IoT Co., Ltd.",
  "dac_root_ca": "-----BEGIN CERTIFICATE-----\n...",
  "schemas": [
    {
      "device_type": "smart-light",
      "version": "1.2.0",
      "schema_url": "https://acme-iot.com/.well-known/ohai/schemas/smart-light.json",
      "schema_hash": "sha256:abcdef..."
    }
  ],
  "updated_at": "2026-03-01T00:00:00Z"
}
```

#### 注册流程（全自动，借鉴 ACME HTTP-01 验证）

```text
┌──────────────┐                              ┌───────────┐
│  厂商         │                              │   OVR     │
│ acme-iot.com │                              │  (注册表)  │
└──────┬───────┘                              └─────┬─────┘
       │                                            │
       │ ① POST /api/v1/register                    │
       │   { vendor_domain, dac_root_ca }           │
       │ ──────────────────────────────────────────>│
       │                                            │
       │ ② 返回 Challenge Token (随机字符串)         │
       │   { token: "gB3k...xQ7f",                  │
       │     expires_in: 300 }                      │
       │ <──────────────────────────────────────────│
       │                                            │
       │ ③ 厂商将 Token 放置到约定路径:              │
       │   https://acme-iot.com/                    │
       │     .well-known/ohai/challenge/gB3k...xQ7f │
       │   文件内容 = token + "." + DAC Root CA      │
       │             公钥的 SHA-256 指纹             │
       │                                            │
       │ ④ POST /api/v1/register/verify             │
       │   { vendor_domain, token }                 │
       │ ──────────────────────────────────────────>│
       │                                            │
       │              ⑤ OVR 主动 GET 厂商域名上的    │
       │                验证文件，校验:               │
       │                a) Token 值匹配              │
       │                b) 公钥指纹与步骤①提交的      │
       │                   DAC Root CA 一致          │
       │                c) Token 未过期 (5 分钟内)    │
       │                → 通过则写入注册表            │
       │                                            │
       │ ✅ 注册完成，无需任何人类审批                 │
```

**详细步骤说明**：

1. **厂商发起注册**：厂商调用 OVR 的注册 API，提交自己的域名和 DAC Root CA 公钥证书。OVR 此时不信任该请求——任何人都可以声称自己拥有 `acme-iot.com`。

2. **OVR 签发 Challenge Token**：OVR 生成一个 URL 安全的随机字符串（至少 128 bit 熵，Base64url 编码，如 `gB3k...xQ7f`），绑定本次注册请求，有效期 5 分钟。Token 通过 API 响应返回给厂商。

3. **厂商放置验证文件**：厂商需要在自己域名的 Web 服务器上创建一个文件，路径为 `/.well-known/ohai/challenge/{token}`。文件内容格式为 `{token}.{dac_root_ca_fingerprint}`，其中指纹为 DAC Root CA 公钥的 SHA-256 哈希（Hex 编码）。**将公钥指纹编入验证文件**防止攻击者截获 Token 后在另一个域名上完成验证——即使攻击者拿到了 Token，也无法伪造与目标 DAC Root CA 匹配的指纹。

4. **厂商通知 OVR 验证**：厂商放置好文件后，调用 OVR 的验证 API 触发校验。这避免了 OVR 盲目轮询，减少不必要的 HTTP 请求。

5. **OVR 执行验证**：OVR 向 `https://{vendor_domain}/.well-known/ohai/challenge/{token}` 发起 HTTPS GET 请求（**必须是 HTTPS**，防止 DNS 劫持下的 HTTP 明文篡改），校验：
   - 响应状态码为 200
   - 文件内容中的 Token 与步骤②签发的一致
   - 文件内容中的公钥指纹与步骤①提交的 DAC Root CA 公钥匹配
   - Token 仍在 5 分钟有效期内
   - 全部通过后，OVR 将厂商信息写入注册表，同步爬取 `/.well-known/ohai/vendor.json` 建立完整记录

::: details 为什么不用 DNS TXT 记录验证？

ACME 协议还支持 DNS-01 验证（在 DNS TXT 记录中放置 Token），这种方式不需要厂商运行 Web 服务器。OHAI 选择 HTTP 验证（类似 ACME HTTP-01）的原因：

1. **厂商本来就需要 Web 服务器**：OHAI 的域名锚定身份要求厂商在 `/.well-known/ohai/vendor.json` 持续发布厂商信息，因此厂商必然运营着 HTTPS 服务。HTTP 验证不引入额外依赖。
2. **更低的操作门槛**：放置一个文件比修改 DNS TXT 记录更简单，特别是对于不直接管理 DNS 的小型厂商。
3. **更快的生效时间**：DNS 记录变更受 TTL 和传播延迟影响（可达数小时），HTTP 文件放置后即时生效。

如果未来出现不运营 Web 服务器的厂商需求（例如仅通过 OVR 直接提交模式注册），可考虑追加 DNS-01 作为补充验证方式。

:::

**注册要求**：
- 拥有一个域名，且该域名支持 HTTPS 访问
- 能在域名下放置验证文件（需控制 Web 服务器）
- 无认证费，无审核委员会，无排队等待

#### 厂商信息更新

厂商可随时更新 `vendor.json` 中的内容（如新增 Schema、更新证书）。OVR 定期爬取已注册厂商的端点，自动同步变更。厂商也可主动调用 OVR API 触发即时同步。

### OVR 数据内容

| 数据 | 来源 | 用途 |
|------|------|------|
| 厂商 DAC Root CA 公钥 | 厂商域名端点 | Server 验证设备 DAC |
| 厂商元数据 (vendor_id, 名称等) | 厂商域名端点 | Console 向用户展示设备来源 |
| 设备 Schema 定义 | 厂商域名端点 | AI 引擎理解设备能力；公众审计 |
| 域名验证记录 | OVR 自动验证 | 证明厂商身份 |
| 安全公告 | 厂商主动发布 | 密钥泄露/漏洞通告 |

### Schema 公开可见性与 AI 安全

设备 Schema 定义了设备的能力描述（属性、命令、状态），是 AI 引擎理解和操控设备的基础。由于 Schema 内容会直接进入大语言模型的上下文，恶意厂商可能在 Schema 中嵌入**提示注入 (Prompt Injection)** 载荷——例如在设备描述字段中植入伪装成系统指令的文本，诱导 AI 引擎执行非预期操作。

OVR 将所有 Schema 定义**公开托管、公众可审计**，通过透明性构建安全防线：

- **公众审计**：安全研究人员和社区成员可审查任意厂商的 Schema 内容，发现可疑的提示注入模式并公开报告。
- **自动化扫描**：OVR 可运行自动化工具扫描新提交或变更的 Schema，检测已知的提示注入模式（如包含 `ignore previous instructions`、`you are now` 等特征字符串的字段值）。检测到可疑内容时自动标记并通知厂商修正。
- **历史可追溯**：Schema 的每次变更都记录在 OVR 的 append-only 日志中，厂商无法悄无声息地修改 Schema 内容。
- **Server 侧纵深防御**：即使 Schema 通过了公开审计，Server 的 AI 引擎仍应将 Schema 内容视为**不可信输入**，在解析时进行沙箱隔离和输出过滤。OVR 的公开透明是纵深防御的一环，而非唯一防线。

### 架构：缓存层而非信任源

```text
              ┌──────────────┐
              │  厂商域名      │  ← 信任的源头 (Source of Truth)
              │  /.well-known │
              │  /ohai/       │
              │  vendor.json  │
              └──────┬───────┘
                     │
          OVR 定期爬取 & 聚合
                     │
              ┌──────▼───────┐
              │     OVR      │  ← 缓存与发现层 (Cache & Discovery)
              │  聚合所有厂商  │     不是信任的源头
              │  的注册信息    │     任何人可运营镜像
              └──────┬───────┘
                     │
              Server 批量同步
                     │
              ┌──────▼───────┐
              │ OHAI Server  │  ← 本地缓存，离线可用
              │ 本地厂商信任库  │
              └──────────────┘
```

**关键特性**：
- **OVR 是缓存，不是权威**：信任的源头始终是厂商域名。OVR 只做聚合和分发。
- **多实例共存**：OHAI 官方运营默认 OVR 实例，任何组织可运营镜像。代码完全开源。
- **Server 可配置多个 OVR 源**：也可直接从厂商域名拉取（作为 fallback）。
- **离线可用**：Server 将厂商信任库缓存到本地，日常设备验证无需联网。

### 运营模式

- **OHAI 官方**运营默认 OVR 实例，代码开源。
- **任何组织或个人**可运营 OVR 镜像，数据通过厂商域名端点独立验证，镜像之间无需互信。
- **Server 默认同步官方 OVR**，可额外配置其他 OVR 实例或直接添加厂商域名。
- **运营成本极低**：OVR 本质上是一个定时爬虫 + 静态文件服务，无需复杂的共识机制或区块链基础设施。

### 数据完整性与可审计性

OVR 维护一份 **append-only 操作日志**，记录所有厂商注册、更新和安全公告事件。日志以 Merkle Tree 组织，任何人可校验完整性：

- 每条日志条目包含：操作类型、厂商域名、时间戳、数据哈希、前一条目哈希。
- OVR 定期发布带签名的 **Tree Head**（Merkle Root + 时间戳），镜像和 Server 可交叉比对。
- 若某个 OVR 实例篡改历史记录，与其他镜像的 Tree Head 比对将立即暴露不一致。

### 安全考量

#### 域名仿冒

若攻击者注册 `philips-smart-home.com` 冒充飞利浦，Console 会向用户展示 "设备来自 philips-smart-home.com"。这是**商标争议**而非协议漏洞，应通过 ICANN 争议解决机制或法律途径处理。OHAI 协议保证"这个设备确实来自该域名"，至于域名是否代表合法品牌，是用户和法律体系的职责——与 HTTPS 证书的信任模型一致。

#### 域名过期

- 设备的 DAC 在出厂时写入 Secure Element，域名变更不影响已有设备。
- Server 本地缓存了厂商 Root CA，不依赖实时域名解析。
- OVR 的 append-only 日志保留历史记录，旧的域名 ↔ Root CA 绑定不会被覆盖。
- 新域名所有者无法伪造旧 Root CA 的私钥签名。

#### 域名劫持

若攻击者劫持厂商域名并替换 `vendor.json` 中的 Root CA：

- 已缓存旧 Root CA 的 Server 不受影响（本地缓存优先）。
- OVR 检测到 Root CA 变更时标记为**待确认**，不自动更新到 Server，需厂商通过额外验证（如旧 Root CA 私钥签名确认轮换）。
- Server 定期同步时对 Root CA 变更执行 **Key Continuity** 检查：若新 Root CA 未经旧 Root CA 的交叉签名确认，则拒绝更新并向 Console 告警。

#### 无域名或无稳定域名的厂商

- OVR 支持**直接提交模式**：厂商直接向 OVR 实例提交 Root CA 和 Schema，不通过域名验证。
- 此类提交在 OVR 中标记为**未经域名验证 (unverified)**，Console 向用户展示降级的信任提示："设备厂商身份未经独立验证"。
- 设备的安全入网流程（SPAKE2+、NOC）不受影响，仅 DAC 真实性缺少域名背书。
