# 历史分析数据保护规范

本文档定义 OHAI 在“基于历史数据的智能建议”场景下，对家庭行为画像、分析依据和建议生成链路的保护要求。

- 范围：适用于设备历史数据的采集、特征提取、画像生成、建议生成、审计记录、删除与导出
- 目标：在保留 OHAI 主动建议能力的前提下，为最敏感的数据处理链条建立明确、可实现、可审计的安全边界
- 定位：本规范补充 [AI 集成](./ai-integration.md#示例-5基于历史数据的智能建议) 中“观察 → 洞察 → 建议 → 确认 → 部署”的流程，回答“如何安全地做画像”而不是“是否允许画像”

## 1. 设计原则

OHAI 要想基于历史行为给出有价值的自动化建议，必须对家庭设备使用模式形成较细粒度的画像。这是系统智能的重要来源，不应被回避。

但画像一旦形成，就不再只是普通运行数据，而是高敏感资产。即使 OHAI Server 运行在家庭内网，系统内部仍然需要回答以下问题：

- 哪些数据会进入分析
- 哪些设备类别默认不参与画像
- 画像结果谁能看见
- 原始历史、推断结论和建议依据保留多久
- 删除请求如何真正生效
- 是否允许将该类数据发送给云端模型或第三方服务
- 建议生成过程如何被追溯和复盘

因此，本规范采用以下原则：

1. 画像能力允许存在，但必须显式开启，不能隐式运行
2. 画像数据视为高敏感数据，保护级别不低于私密聊天内容
3. 不仅保护原始历史数据，也保护模型推断出的画像结论和建议依据
4. 本地部署可以降低外泄风险，但不能替代最小化、访问控制、保留期和审计
5. 高敏感类别默认从分析中排除，只有用户明确放宽后才可参与

## 2. 威胁模型

本规范防御的不只是“数据出公网泄露”，还包括以下风险：

1. 系统内部误用：Console 页面、家庭共享成员、插件、调试工具、备份系统、导出功能看到超出其职责范围的画像数据
2. 推断升级：单个设备事件本身不敏感，但组合后可以稳定推断作息、离家规律、睡眠、洗澡、是否有老人儿童宠物等高敏感事实
3. 长期留存：历史数据、画像摘要、建议依据和审计记录长期积累，导致未来暴露成本升高
4. 误判驱动错误自动化：模型将偶发模式误判为稳定习惯，持续生成不合适的建议
5. 第三方扩面：未来切换云模型、远程运维、第三方分析器或家庭备份系统后，历史画像暴露面被扩大

## 3. 数据分级与对象模型

OHAI 在历史分析链路中至少应区分以下四类数据对象，并分别保护：

### 3.1 原始历史数据

指用于分析的设备状态变化、事件记录、命令执行记录和环境数据采样。

示例：

- `water_heater_01` 在 06:53 开始加热、07:11 停止
- 卧室灯在 23:17 熄灭
- 门磁在 08:12 关闭，18:44 打开

要求：

- 必须记录来源设备、能力、时间戳、采样粒度和采集目的
- 不得因为“可能以后有用”而无限制纳入其他无关数据
- 不得默认把全屋所有设备历史都纳入分析

### 3.2 派生画像数据

指由模型或统计模块从原始历史中推导出的习惯、规律、置信度、时间窗口和场景标签。

示例：

- “过去 14 天中有 12 天在 06:50-07:10 使用热水”
- “工作日 08:30-18:00 长时间无人”
- “卧室灯和空调通常在 23:00 后进入夜间模式”

要求：

- 派生画像必须单独存储，不能与普通运行日志混放
- 派生画像的敏感等级不得低于原始历史数据
- 派生画像必须附带时间窗、算法版本、置信度和来源摘要

### 3.3 建议依据快照

指系统在生成一条具体建议时保留下来的最小必要解释材料，用于用户查看和事后审计。

示例：

- 时间窗：最近 14 天
- 参与设备：热水器、水流量传感器
- 模式摘要：12/14 天在 07:00 左右用热水
- 置信度：0.91

要求：

- 默认只保存摘要，不保存完整原始序列副本
- 必须能够关联到最终建议文本、用户确认记录和部署出的规则版本

### 3.4 审计记录

指谁在何时触发了分析、访问了画像、启用了云模型、删除了画像、批准了建议。

要求：

- 审计记录必须防篡改追加写
- 审计记录不得包含超出审计目的的原始生活细节
- 审计记录与画像数据应逻辑隔离存储

## 4. 默认策略

### 4.1 历史分析默认关闭

OHAI Server 对“基于历史数据的智能建议”能力必须默认关闭，只有用户显式开启后才允许启动分析任务。

显式开启至少包含：

1. 说明用途：仅用于生成自动化建议，不用于无关目的
2. 说明参与范围：哪些设备类别或房间会被纳入
3. 说明模型位置：本地模型或云模型
4. 说明保留期：原始历史、画像摘要、审计记录的保存时间
5. 说明撤销方式：如何暂停、清除和重新开启

### 4.2 高敏感类别默认排除

以下类别必须默认不参与历史画像分析，除非用户完成额外确认：

- 门锁、门禁、撤防/布防
- 卧室相关设备
- 卫浴相关设备
- 室内摄像头、麦克风、门铃视频
- 老人护理、儿童看护、医疗辅助类设备

额外确认必须明确说明：

- 纳入该类别会推断什么类型的生活规律
- 一旦启用，建议与画像摘要可能暴露更强的私人信息

### 4.3 本地优先，云端二次确认

历史画像分析应采用“本地优先”策略：

- 本地模型可用时，默认使用本地模型
- 云模型必须默认关闭
- 若启用云模型，必须单独进行二次确认，而不能复用“开启历史分析”的授权

云模型启用前，系统必须展示：

- 服务提供方名称
- 传输的数据范围和粒度
- 是否发送原始序列、聚合特征还是仅发送摘要
- 数据保留与训练策略
- 如何随时撤销

## 5. 数据最小化实施方法

### 5.1 设备范围最小化

分析任务必须先根据目标场景构建设备白名单，再读取历史数据，而不是先抓取全量历史再交给模型筛选。

示例：

- 识别“晨间热水”模式时，只读取热水器、水流量、相关温度传感器
- 识别“离家节能”模式时，只读取门磁、安防布防状态、空调和主要照明负载

### 5.2 时间窗最小化

系统必须为每次分析指定最小必要时间窗，默认不得无限制回看长期历史。

建议默认值：

- 常规行为建议：14 天窗口
- 周期性强的模式：30 天窗口上限
- 超过 30 天的历史分析必须要求显式说明原因

### 5.3 粒度最小化

能用聚合特征完成判断时，不应把逐秒原始序列交给模型。

建议流程：

1. 先由 Server 侧统计模块提取候选特征，例如时间段频率、持续时长、周内分布、触发次数
2. 仅将必要特征和少量上下文摘要交给模型解释和生成建议
3. 只有在统计特征不足以表达模式时，才允许读取更细的原始序列

### 5.4 房间与场景隔离

若建议只涉及某一生活场景，系统应优先限制在对应房间或设备组内分析，不得跨房间拼接无关数据。

## 6. 存储与加密实施方法

### 6.1 分层存储

至少应分为三类逻辑存储：

1. 原始历史存储
2. 派生画像与建议依据存储
3. 审计日志存储

三类数据不得直接混存在同一张无隔离的日志表中。

### 6.2 静态加密

历史分析相关数据必须在磁盘静态加密。推荐实现：

- 为每个 Home 生成独立的数据加密密钥（DEK）
- 使用 Server 主密钥或操作系统密钥链保护 DEK
- 原始历史、画像摘要、建议依据分别使用独立的数据域标识进行加密

### 6.3 内存暴露控制

分析任务在内存中展开敏感历史时，必须限制：

- 单次任务读取的最大记录数
- 中间缓存的存活时间
- 调试日志中打印原始记录的能力

默认要求：

- 生产模式下禁止把原始历史数据写入普通调试日志
- 错误日志中只允许出现任务 ID、设备类别、窗口长度和异常类型

## 7. 访问控制实施方法

### 7.1 角色最小权限

Console App 至少应区分以下能力：

- 查看建议
- 查看建议依据摘要
- 查看完整画像详情
- 修改分析范围
- 清除画像数据
- 启用云模型

推荐默认权限：

- Owner：全部允许
- Admin：允许查看建议与依据摘要，不默认允许启用云模型或查看高敏感画像详情
- Member：仅查看面向自己的建议，不查看完整画像结论
- Guest：不允许访问画像相关页面

### 7.2 按敏感级别遮罩

即使用户有查看建议权限，系统也不应默认展示全部分析细节。

示例：

- 普通页仅展示“过去两周存在稳定晨间热水模式”
- 详细页才展示“12/14 天，06:50-07:10，用水持续 15-25 分钟”

### 7.3 插件与外部接口限制

第三方插件、Webhook、导出接口默认不得访问派生画像数据与建议依据快照，除非用户单独授权。

## 8. 审计与可追溯实施方法

每次分析任务、每条建议和每次用户确认，至少应记录以下字段：

| 字段 | 说明 |
|---|---|
| `analysis_job_id` | 本次分析任务 ID |
| `trigger_type` | 定期分析、手动触发、模型重跑 |
| `time_window` | 参与分析的历史时间窗 |
| `device_categories` | 参与的设备类别 |
| `device_ids_hash` | 参与设备的稳定散列标识 |
| `feature_summary` | 输入模型的摘要特征 |
| `model_location` | 本地或云端 |
| `model_id` | 模型标识和版本 |
| `policy_snapshot` | 当时生效的数据治理策略快照 |
| `confidence` | 模式识别置信度 |
| `suggestion_text` | 提供给用户的建议文本 |
| `user_decision` | 接受、拒绝、稍后处理 |
| `generated_rule_id` | 若用户接受，对应生成的规则 ID |
| `generated_rule_version` | 规则版本或内容摘要 |
| `simulation_report_id` | 部署前模拟测试报告 |
| `operator` | 发起或批准该操作的主体 |
| `timestamp` | 记录时间 |

附加要求：

1. 审计记录必须可导出，便于排查“系统为什么推荐了这条规则”
2. 审计记录中的 `feature_summary` 应优先使用摘要而非原始序列
3. 若启用了云模型，审计中必须额外记录出站目的地和数据范围

## 9. 删除、暂停与保留期实施方法

### 9.1 一键暂停

系统必须支持“一键暂停历史分析”。暂停后：

- 不再启动新的分析任务
- 不再为画像用途继续累积新的特征缓存
- 已存在的规则不受影响，但不得继续根据新历史生成新的主动建议

### 9.2 一键清除画像依据

系统必须支持“一键清除画像依据”，至少删除：

- 派生画像数据
- 建议依据快照
- 待处理建议队列

默认不要求同时删除通用设备运行历史；但若该运行历史被单独标记为“仅供画像分析”，则也应被删除。

### 9.3 保留期

建议默认保留期：

- 原始分析缓冲历史：14 天
- 派生画像数据：30 天
- 建议依据快照：30 天
- 审计记录：90 天

实现要求：

- 保留期到达后必须自动删除，而不是仅标记过期
- 用户手动清除后，删除动作应在 24 小时内完成
- 若存在离线备份，文档必须说明备份副本的清除延迟

## 10. 云模型与第三方服务边界

启用云模型或第三方分析服务时，必须额外满足：

1. 不得发送默认排除类别的原始历史，除非用户单独对该类别再次授权
2. 优先发送聚合特征和脱敏摘要，避免直接发送可重建生活时间线的原始事件流
3. 必须支持随时关闭，并在关闭后停止新的出站请求
4. 必须在审计中记录每次出站分析的目的地、数据范围和任务 ID
5. 若服务方声明可保留数据用于训练或质量改进，必须单独征得用户确认

## 11. UI 与产品交互要求

Console App 对历史分析相关能力至少应提供以下用户界面：

1. 历史分析总开关
2. 参与分析的设备类别选择器
3. 高敏感类别的单独确认入口
4. 本地模型 / 云模型状态展示
5. 最近一次分析任务的摘要与时间窗
6. 每条建议的“依据摘要”查看入口
7. 一键暂停
8. 一键清除画像依据
9. 审计导出入口

建议文案要求：

- 不使用模糊表述，例如“帮助提升体验”
- 直接说明系统会分析历史行为模式，并可能推断作息与生活规律

## 12. 与 AI 集成的衔接要求

实现 [AI 集成](./ai-integration.md#示例-5基于历史数据的智能建议) 时，必须补齐以下约束：

1. `API.query_history/4` 不应成为全量历史自由读取接口；历史建议任务应先经过设备白名单、时间窗和粒度裁剪
2. 生成建议时，必须同步生成建议依据摘要，而不是仅输出自然语言建议
3. 用户确认建议并部署规则前，应将建议依据摘要与模拟报告一起展示
4. 若分析任务使用云模型，应在建议卡片中明确标记“由云端分析生成”

## 13. 最低合规结论

若某个 OHAI 实现满足以下条件，才可宣称其“安全支持基于历史数据的智能建议”：

1. 历史分析默认关闭，且显式开启
2. 高敏感类别默认排除
3. 本地优先，云端单独授权
4. 原始历史、派生画像、审计记录分层存储并静态加密
5. 具备角色化访问控制与敏感细节遮罩
6. 每条建议都可追溯到时间窗、设备类别、模型版本和策略快照
7. 具备一键暂停和一键清除画像依据能力

未满足上述最低条件的实现，不应默认启用主动历史分析建议。
